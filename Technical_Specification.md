## 3. Data Flows

### A. Telemetry Processing Pipeline
1. Ingestion: External agents or simulation endpoint posts a raw log to `POST /api/telemetry`.
2. Parsing: A Regex engine parses the log to extract `timestamp`, `source_asset`, `user_principal`, and `bytes_transferred`.
3. Feature Engineering:
   - `timestamp` is parsed into `hour_of_day` and `day_of_week`.
   - `source_asset` and `user_principal` are mapped to categorical index features.
   - `bytes_transferred` is scaled.
4. Anomaly Detection: The custom pre-trained `IsolationForest` model evaluates the feature array.
   - If the anomaly score is less than the calculated threshold (indicating an outlier), the status is set to `CRITICAL_ANOMALY`.
5. Persistence: The record (log details + score + status) is committed to Supabase (or SQLite fallback).
6. Alerting: The backend broadcasts the result to all active WebSocket clients.

### B. Micro-Isolation Loop
1. Detection or Manual Trigger: An anomaly is detected or the user clicks "Isolate Node" in the dashboard.
2. Mitigation Command: The frontend invokes `POST /api/isolate-node` with the node's identifier.
3. Execution: The backend runs a PowerShell (`.ps1`) or Shell (`.sh`) script depending on OS.
   - The script performs localized isolation actions (e.g. updating firewall tables, stopping simulated docker containers, updating the db state node to `ISOLATED`).
4. Broadcast: The state change is saved to the database and broadcasted via WebSocket.
5. UI Update: The node style on the Vis.js topology graph changes dynamically (color shifts, pulsing rings stop/turn red, containment ledger registers isolation timestamp and MITRE ATT&CK vectors).

## 4. Database Schema (schema.sql)

```sql
-- Schema Migration: Telemetry and Isolation logs
CREATE TABLE IF NOT EXISTS telemetry_logs (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    timestamp TIMESTAMPTZ NOT NULL,
    source_asset VARCHAR(255) NOT NULL,
    user_principal VARCHAR(255) NOT NULL,
    bytes_transferred INTEGER NOT NULL,
    anomaly_score FLOAT NOT NULL,
    status VARCHAR(50) NOT NULL
);

-- Performance Indexes
CREATE INDEX IF NOT EXISTS idx_telemetry_source_timestamp 
ON telemetry_logs (source_asset, timestamp DESC);
```

## 5. Front-End State Management

- **WebSockets (`WS_Client`)**: Maintains a persistent connection to `ws://localhost:8000/api/ws`. Listens for raw telemetry logs and system alerts.
- **Topology State**: Holds the nodes and edges for the Vis.js network. Node properties (color, shadow, label) are updated dynamically when an alert is received.
- **Threat Level State**: Tracks the overall status (`SECURE`, `WARNING`, `CRITICAL`) based on recent anomalies (e.g., if a CRITICAL_ANOMALY was received in the last 15 seconds, threat level climbs to Red/Breach).
- **Containment Ledger State**: A reactive list tracking which nodes have been isolated, showing the isolated asset, the user principal involved, the byte size of the malicious payload, and corresponding MITRE ATT&CK tags (e.g. *T1048: Exfiltration Over Alternative Protocol* or *T1486: Data Encrypted for Impact*).
